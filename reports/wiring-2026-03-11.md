## Detailed Findings

### 1. 🔴 CRITICAL — Agent Registry Never Bootstrapped

**WHERE**: `src/agents/registry.ts` defines `bootstrapAgents()` which registers all 35 agents into `agentRegistry` Map.

**WHAT'S LOST**: `bootstrapAgents()` is only called in test files (`src/agents/__tests__/agent-smoke.test.ts:85`). No startup hook in server, worker, or app initialization ever calls it.

**WHY IT MATTERS**: The entire agent-delegate resolution chain (`src/server/workflows/agent-delegate.ts:136-157`) falls back to `agentRegistry.get(agentName)` — which returns `undefined` at runtime because the registry is empty. Workflow steps with `type: agent-delegate` will fail to resolve any agent.

**IMPACT**: All YAML workflow agent delegation is broken in production.

---

### 2. 🔴 CRITICAL — Cron Workflows Parsed But Never Scheduled

**WHERE**:
- Schema: `src/server/workflows/workflow-schema.ts:73-76` defines `cronTrigger` with schedule field
- Validator: `src/server/workflows/validator.ts:129-134` validates cron syntax (5-6 fields)
- Missing: No `boss.schedule()` call anywhere for cron workflows

**WHAT'S LOST**: YAML workflows with `trigger.type: "cron"` are parsed and validated correctly, but no code ever registers them as pg-boss scheduled jobs. The `WORKFLOW_QUEUE` constant (`workflow:run`) is defined but has no consumer handler in `src/server/worker/jobs/index.ts`.

**WHY IT MATTERS**: Scheduled workflows (daily-pipeline, engagement-sweep) will never fire automatically.

### 3. 🔴 CRITICAL — Workflow Queue Has No Consumer

**WHERE**: `src/server/api/routers/workflows.ts:109` sends to `workflow:run` queue via `boss.send()`.

**WHY IT MATTERS**: Manual workflow triggers from the dashboard silently do nothing.

### 6. 🔴 CRITICAL — Cookie Shape Mismatch Between Services

**WHERE**:

- Consumer: `src/server/workers/competitor-worker.ts:22-26` — expects `cookies: Record<string, string>`

**WHAT'S LOST**: Scraper pool sends structured cookie objects; competitor worker expects a flat key-value map. The consumer will receive the wrong shape.

**WHY IT MATTERS**: Competitor scraping pipeline will fail or produce incorrect results when processing cookies.

### 7. 🟠 HIGH — Worker Job Handler Bypasses LLM Budget

**WHERE**:
- ✅ Budget checked: `src/server/workflows/executor.ts:299` — `checkLlmBudget()` before agent-delegate steps
- ❌ Budget NOT checked: `src/server/worker/jobs/handlers/agent-execute.ts:16` — calls `executeAgentDelegate()` directly without `checkLlmBudget()`

**WHAT'S LOST**: Agents invoked via standalone pg-boss jobs skip budget enforcement. Budget is tracked post-call (`trackLlmSpend()` works) but the pre-flight gate is missing.

**WHY IT MATTERS**: Orgs can exceed their daily LLM budget via the job queue path. The `workflow:run` path is gated, but direct `agent-execute` jobs are not.

### 8. 🟠 HIGH — 3 Posting Paths Bypass Circuit Breaker

**WHERE**:
- ✅ Checked: `src/server/services/distribution-scheduler.ts:109` — calls `canPost()` before scheduling
- ❌ Not checked: `src/server/workers/post-worker.ts:38` — calls `postContent()` directly
- ❌ Not checked: `src/server/worker/jobs/handlers/content-publish.ts:64` — calls `postContent()` directly
- ❌ Not checked: `src/server/worker/jobs/handlers/content-schedule.ts:64` — enqueues without `canPost()`

**WHAT'S LOST**: These 3 paths filter `circuitState: { not: "OPEN" }` at query time but don't call `canPost()` at execution time. Circuit state can change between query and execution. HALF_OPEN trial logic is never applied.

**WHY IT MATTERS**: Posts can be attempted against accounts with OPEN circuit breakers if state changes after the initial DB query. The HALF_OPEN → trial → success/fail flow only works through distribution-scheduler.

### 9. 🟡 MEDIUM — Scraper Pool Proxy Architecture Break

**WHERE**:
- Browser accounts: `src/server/services/browser-helpers.ts:47-52` — fetches proxy URL from Infisical via `infisicalProxyPath`
- Scraper pool: `services/scraper-pool/src/proxy-manager.ts:24-35` — loads from `PROXY_RESIDENTIAL_ENDPOINT` env var at startup, caches in-memory indefinitely

**WHY IT MATTERS**: Inconsistent security posture. If the scraper-pool container is compromised, proxy URLs are exposed in memory. Browser account proxies are properly protected via Infisical.

## Systemic Patterns

### Pattern 1: "Built But Never Wired"
The dominant pattern across this codebase. Services, agents, safety hooks, and infrastructure modules are fully implemented but never imported or called from orchestration layers.

**Affected**: R2 storage (10 functions), SSE broadcasting, notifications, safety hooks, output schemas, brand loader, tool wrappers, 20+ agents, metrics module.

**Root cause**: Implementation happened bottom-up (build services first), but top-down wiring (routers/workers importing services) was never completed.

### Pattern 2: "Works in Path A, Not Path B"
Critical safety checks are enforced in the "happy path" but bypassed in alternative execution paths.

| Check | Path A (enforced) | Path B (bypassed) |
|-------|-------------------|-------------------|
| LLM Budget | Workflow executor | Worker agent-execute handler |
| Circuit Breaker | Distribution scheduler | Post worker, content-publish, content-schedule |
| Agent Registry | Test suite | Runtime (never bootstrapped) |

### Pattern 3: "No Shared Contract Between Services"
Types are defined independently in main app and microservices with no shared package. This causes:
- Cookie shape mismatch (scrape:result)
- Transform type mismatch (media:task)
- Missing job types (audio-check)

### Pattern 4: "Infisical for Some, Env Vars for Others"
Credential management is inconsistent: browser account proxies use Infisical properly, but scraper-pool loads proxy URLs from env vars.
